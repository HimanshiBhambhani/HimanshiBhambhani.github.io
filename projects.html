ChatGPT Voice Input Redesign

Product Discovery · User Research · Experimentation — 2026
Live Transcription with Review — making voice feel as controlled as typing

Mobile-first Indian professionals avoid voice input because capture is a black box — no transparency, no way to catch errors, no control before sending. I ran the discovery, framed the problem, designed the flow, and defined the experiment to prove it.

  1. Landscape & UX audit

    Mapped India's voice-first context (950M+ mobile users, oral-first economy) and benchmarked ChatGPT against WhatsApp, Gemini & Claude on latency, web-search flow, Hinglish prosody, and interruption handling.Competitive teardown · market sizing

  2. User research (n=47)

    Quantified the barriers: 36% blind-input trust gap, 32% efficiency paradox, 26% social/environmental friction, 17% accent anxiety — then pulled the mental models behind them.Survey · interviews · thematic coding

  3. Segmentation & problem framing

    Scored five segments; chose mobile-first early-career professionals (18–35). Distilled a single problem statement via a Problem Framing Canvas.Impact scoring · personas · framing canvas

  4. Solution design

    Generated three mutually-exclusive directions (see live / hear back / control at the end) and chose Live Transcription with Review — solid streaming text in an editable field before send.Figma wireframes · 4-screen flow

  5. Experimentation plan

    Defined the hypothesis, primary & guardrail metrics, and a decision rule — ship on significant completion + repeat-usage lift; iterate on high toggle-off; kill on latency regression.A/B design · KPI tree · metric instrumentation

🧠 Key product decisions

  • Transparency over accuracy: live transcription makes errors catchable — a UI fix, not a model retrain.
  • Deliberate send over auto-send: one extra tap buys full user control and trust.
  • Dictation-only in v1: keeps the experiment isolated and clean; ghost-text confidence highlighting descoped to Phase 2.

📊 What I measured

  • Primary: voice completion rate, mic-button CTR, 7-day repeat voice usage.
  • Secondary: post-transcription edit rate (>20% target confirms the review step is used).
  • Guardrails: STT latency <200ms, toggle-off rate <15%, crash rate.
Key learning

"Users don't distrust voice — they distrust invisibility during capture. The trust gap was psychological, not technical. Live transcription already exists on every iPhone and in Google Docs; ChatGPT was simply the outlier for not surfacing it. The best insight of the project wasn't a new feature — it was reframing the problem so the solution became obvious."

  1. Problem definition & data

    Scoped the recommendation problem and built the data layer — loading and normalizing 21,000+ restaurants from a public dataset, with EDA to shape the schema.Python · Pandas · EDA

  2. Models & filtering engine

    Designed typed UserPreferences / Recommendation dataclasses and a filtering engine with fuzzy matching (configurable threshold) and 5-stage progressive relaxation when results run thin.fuzzywuzzy · dataclasses · 60+ unit tests

  3. LLM integration

    Abstracted the LLM behind a provider interface (Groq / OpenAI / Ollama) and engineered prompts that rank candidates and explain each pick in plain language.Groq · Llama 3.3 70B · prompt engineering

  4. Safety & reliability

    Hardened for live traffic: input validation & SQL-injection prevention, output-safety guardrails, and rate limiting — backed by 35+ automated tests.input guardrails · rate limiting

  5. Deployment

    Shipped the full product on Vercel + Railway with Docker support and a documented, repeatable deploy.Vercel · Railway · Docker

🧠 Key product decisions

  • Progressive relaxation: never return an empty screen — loosen filters in stages so users always get a usable answer.
  • Provider abstraction: swap Groq / OpenAI / Ollama without touching product logic — avoids lock-in.
  • Budget tiers: map raw prices into ₹0–500 / ₹500–1500 / ₹1500+ so users choose by intent, not numbers.

✓ Key results

  • Structured six-phase delivery from concept to live deployment.
  • 60+ unit tests on the engine, 35+ automated tests overall.
  • Input-safety guardrails + rate limiting validated on live traffic.
Key learning

"Shipping an AI product is mostly the un-glamorous 90% — the fallback when there are no results, the guardrail on a malicious input, the rate limit at 2am. The model is the easy part; making it trustworthy and reliable enough to put in front of real traffic is the actual product work."

Stack: Python · Pandas · fuzzywuzzy · Groq (Llama 3.3 70B) · Streamlit · Vercel · Railway · Docker

Mutual Fund FAQ Assistant

Full-Stack RAG Product with Safety Guardrails — 2025
Grounded answers on mutual funds, with the guardrails finance demands

A retrieval-augmented chatbot that answers mutual-fund FAQs from a live, self-refreshing knowledge base — built full-stack, with multi-layer guardrails so it informs without ever giving financial advice.

  1. Knowledge base ingestion

    Built a scraping + chunking pipeline over 60 fund pages, embedded with Gemini into a 437-chunk ChromaDB index — and scheduled a daily APScheduler cron to keep it current.APScheduler · Gemini embeddings · ChromaDB

  2. Retrieval + generation

    Wired vector search to a Llama 3.3 generator so every answer is grounded in retrieved chunks rather than the model's memory — reducing hallucination on a domain where accuracy matters.RAG · Llama 3.3 · vector search

  3. Multi-layer guardrails

    Layered three safety checks: PII detection on inputs, advisory refusal (informs but won't advise "should I buy?"), and off-topic filtering — each with a graceful fallback response.PII detection · policy prompts · fallbacks

  4. Full-stack delivery

    Shipped a FastAPI backend on Railway with a Next.js frontend, deployed on Vercel — a complete, usable product, not a notebook.FastAPI · Railway · Next.js · Vercel

🧠 Key product decisions

  • Advisory refusal by design: in finance, the safest answer to "should I invest?" is a boundary, not a guess.
  • Daily re-scrape: fund data goes stale fast — freshness is a feature, automated via cron.
  • Grounding over fluency: retrieval-first so answers trace back to a source, not the model's imagination.

✓ Key results

  • 437-chunk retrieval index, self-refreshing daily.
  • Three independent guardrail layers with fallbacks.
  • End-to-end deployed full-stack (FastAPI + Next.js).
Key learning

"For a finance assistant, the guardrails are the product. Getting a plausible answer out of an LLM is trivial; knowing when the product should refuse to answer — and doing it gracefully — is what makes it safe enough to actually ship in a regulated domain."

Stack: FastAPI · ChromaDB · Gemini embeddings · Llama 3.3 · APScheduler · Next.js · Railway · Vercel

  1. Product scoping

    Framed the job-to-be-done: stakeholders drown in raw reviews and need a weekly signal, not a spreadsheet. Scoped a repeatable "pulse" cadence.product scoping · JTBD

  2. Ingestion & theme analysis

    Built the flow to ingest up to 1,500 reviews and cluster them into recurring themes and sentiment shifts that a busy stakeholder can absorb in a minute.ingestion · theme extraction

  3. Report generation

    Generated a tight one-page summary — the top themes, what changed week-over-week, and the signals worth acting on.summarization · report gen

  4. MCP delivery + safeguards

    Delivered the report over the Model Context Protocol, and validated on live data with PII scrubbing, idempotency, and audit logging before calling it production-ready.MCP · PII scrubbing · idempotency · audit logs

🧠 Key product decisions

  • One page, on purpose: the product's value is compression — a report nobody reads has zero impact.
  • Idempotent by design: re-running a week never double-counts or corrupts — trust in the numbers is the whole point.
  • MCP as the delivery rail: insight lands where stakeholders already work, not in yet another dashboard.

✓ Key results

  • End-to-end pipeline from ingestion to delivery.
  • Validated on live data before production.
  • PII scrubbing + audit logging for safe handling.
Key learning

"The hard part of an insights product isn't generating the summary — it's earning trust in it. Idempotency, PII scrubbing, and audit logging aren't back-office chores; they're what lets a stakeholder act on the one-pager without second-guessing where the numbers came from."

Stack: Python · Model Context Protocol (MCP) · theme analysis · PII scrubbing · audit logging
